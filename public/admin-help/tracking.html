2. GTM 컨테이너 완료

컨테이너 ID: GTM-N3HSNZPJ

코드 박는 위치는 이미 끝났습니다. 앞으로는 GTM 웹 UI(tagmanager.google.com)에서 태그·트리거·변수만 추가하면 즉시 사이트에 반영됩니다.

코드 측 셋업 (이미 완료)

  • components/seo/GoogleTagManager.tsx — script + noscript iframe 한 번에 처리
  • app/layout.tsx — <body> 최상단 마운트
  • lib/seo.ts — GTM_ID 상수 (env NEXT_PUBLIC_GTM_ID로 override 가능)
  • 어드민 자동 차단: usePathname()이 /admin/*이면 GTM 렌더 0. → 직원 내부 사용이 GA 통계 오염시키지 않음.
✅ 라이브 검증 통과 — www.ozlabpay.kr 에서 dataLayer, google_tag_manager['GTM-N3HSNZPJ'], noscript iframe 모두 정상 동작 확인됨.

GTM 컨테이너 바꿔야 할 때

  1. Vercel 프로젝트 Settings → Environment Variables → NEXT_PUBLIC_GTM_ID 값 변경
  2. Vercel Deployments → 최신 배포 우측 ⋯ → Redeploy
  3. 약 1분 후 사이트에서 새 컨테이너 ID 박힘 확인 (Chrome 확장 Tag Assistant)

3. GA4 (Google Analytics 4) 대기

GA4는 GTM 안에서 설정 태그 1개만 추가하면 됩니다. 코드 변경 0.

1

GA4 프로퍼티 생성

analytics.google.com → 관리(좌하단 톱니) → 프로퍼티 만들기 → 오즈랩페이. 측정 ID G-XXXXXXXXXX 복사.

2

GTM 안에 GA4 설정 태그

GTM GTM-N3HSNZPJ 진입 → 태그 → 새로 만들기 → 태그 유형 "Google 태그" → 태그 ID에 G-XXXXXXXXXX 입력 → 트리거 "Initialization - All Pages" → 저장.

3

미리보기 모드로 검증

GTM 우상단 "미리보기" → URL https://www.ozlabpay.kr 입력 → Tag Assistant 새 탭 열림 → Tags Fired 영역에 "Google 태그" 보이면 OK.

4

GA4 DebugView 확인

GA4 좌측 메뉴 "관리 → 데이터 표시 → DebugView" 진입. 사이트 새로고침하면 실시간 이벤트 흐름 보임. page_view 들어오면 끝.

5

제출 + 게시

GTM 우상단 "제출" → 버전 이름 "GA4 연결" → "게시 및 만들기". 이후로 모든 방문자가 GA4에 카운트됨.

💡 전환 측정은 GA4 "관리 → 이벤트" 메뉴에서 form_submit, generate_lead 등을 전환으로 표시를 켜면 자동 집계 시작. (단, 그 이벤트가 dataLayer에 push 되고 있어야 함 → 10번 섹션)

6. Google Ads 전환 추적 대기

광고비 ROAS 측정의 핵심. 전환 = 상담 신청 폼 제출로 잡습니다.

1

Google Ads 전환 액션 생성

Google Ads → 도구 → 전환 → +새 전환 액션 → "웹사이트" → URL 입력 → "수동으로 전환 액션 만들기" → 카테고리 "리드", 가치 "각 전환마다 같은 값 사용" → 1건당 기여 가치 입력(예: 5만원).

전환 ID 두 개 메모: conversion_id (AW-XXXXXXXXX), conversion_label (영문/숫자 조합).

2

GTM 안에 전환 태그

GTM → 태그 → 새로 만들기 → "Google Ads 전환 추적" → 위에서 받은 conversion_id · conversion_label 입력.

3

트리거

맞춤 이벤트 트리거 → 이벤트 이름 generate_lead → 저장. (이 이벤트는 dataLayer push로 들어옵니다 — 10번 섹션)

4

Ads-GA4 연결 (보너스)

GA4 관리 → 제품 링크 → Google Ads 링크 → 광고 계정 선택. 이러면 광고 → GA4 전환 연결이 자동 import 되어 Ads에서 직접 잡지 않아도 됩니다.

✅ Conversions API(CAPI)는 이미 서버 측에 박혀있음 — env(`META_PIXEL_ID` + `META_CAPI_TOKEN`)만 채우면 즉시 동작. 폼 제출 시 Lead, 매출 입력 시 Purchase 둘 다 서버에서 발사. 픽셀과 event_id 로 자동 dedupe.

7-5. 픽셀 ↔ CAPI dedupe (필수 셋업)

듀얼 발사하면 같은 사용자 행동이 1건이어야 하는데 2건으로 잡힐 수 있음. event_id 매칭으로 Meta 가 자동 dedupe.

  • Lead: 폼 제출 응답 후 generate_lead dataLayer 의 lead_id 값 = consultations.id
  • GTM 안 Meta Pixel 'Lead' 태그에 eventID 변수로 {{DLV - lead_id}} 사용
  • CAPI 는 동일 lead_id 를 event_id 로 보냄 → Meta 자동 dedupe
  • Purchase: 매출 시점엔 브라우저 픽셀이 발사 불가 (어드민 페이지). CAPI 단독으로 충분.

7-6. CAPI env 설정 (대웅 작업)

  1. 비즈매니저 → 이벤트 관리자 → 픽셀 선택 → 설정 → Conversions API → 액세스 토큰 생성
  2. Vercel 프로젝트 ozlab → Settings → Environment Variables → 추가: 
    META_PIXEL_ID = 000000000000000
META_CAPI_TOKEN = EAA...
  3. (선택) 검증 단계엔 META_CAPI_TEST_CODE=TEST12345 추가 → prod 데이터 안 쌓이고 비즈매니저 "테스트 이벤트" 탭에 들어옴
  4. 재배포 → 매출 1건 시뮬레이션 → 비즈매니저 → 이벤트 관리자 → 픽셀 → 개요에서 Purchase 이벤트 확인

10. dataLayer 표준 이벤트 명세 코드작업 대기

위 6·7·8·9번에서 트리거로 쓰는 이벤트 이름들은 아직 사이트에서 push되지 않습니다. 다음 단계 코드 작업(=옵션 A)에서 박을 예정.

아래는 박을 표준 이벤트 명세 (GTM 트리거 만들 때 동일한 이름 쓰면 됨):

이벤트 이름언제 발생데이터 payload
generate_lead GA4 표준 상담 신청 폼 제출 성공 시 cta_id, cta_label, utm_source/medium/campaign, value (1건당 기여 가치)
form_start 폼 첫 필드 포커스 시 cta_id, form_location (hero/floating/modal 등)
cta_click CTA 버튼 클릭 (폼 안 띄우는 단순 anchor 포함) cta_id, cta_label, page_path
phone_click 전화번호 tel: 링크 클릭 page_path, phone_number
kakao_chat_click 카카오톡 채널 상담 버튼 클릭 page_path
scroll_depth 페이지 25/50/75/100% 스크롤 도달 depth_pct, page_path
ℹ️ 왜 generate_lead를 쓰나: GA4 추천 이벤트(공식 doc) 중 하나라 GA4가 자동으로 전환 의도로 인식. 메타 픽셀의 Lead, Google Ads 의 "리드" 카테고리와도 1:1 매핑.

10-B. 전환가치 동적 매핑 (핵심) 코드작업 대기

일반 가이드의 "1건당 5만원 고정값"은 ROAS를 왜곡합니다. 오즈랩페이는 건마다 매출·사은품·net 마진이 다름. 진짜 net 마진(매출 - 사은품 지출)이 GA·광고로 흘러야 정확한 매체별 ROAS가 보입니다.

ℹ️ net 마진 정의 (이번 프로젝트 기준): net_value = revenue_records.amount − COALESCE(gift_cost, 0)
→ 실제 사업 기여 가치. 광고 매체에 이 값이 들어가야 효율 정확함.

10-B-1. 시점이 두 개라는 사실 인지

시점위치가능한 값액션
① 상담 신청 시점 방문자 브라우저 매출 모름 (아직 안 났으니) dataLayer push generate_lead with 추정 평균 net 마진
② 매출 입력 시점 어드민 (방문자 세션 종료됨) 실제 net 마진 확정 서버에서 GA4 Measurement Protocol 호출 — 실제 net 값으로 보정

10-B-2. ① 폼 제출 시점 — 추정값 push

window.dataLayer.push({
  event: 'generate_lead',
  lead_id: 'consult_abc123',    // consultations.id (서버 응답)
  cta_id: 'hero_apply',
  value: 30000,                    // 추정 평균 net 마진 (운영하면서 갱신)
  currency: 'KRW',
  utm_source: 'naver',
  utm_medium: 'cpc',
  utm_campaign: 'spring_launch',
});

lead_id는 나중에 매출 보정 시 두 이벤트를 연결하는 핵심 키. 반드시 consultations.id 와 같은 값.

10-B-3. ② 매출 입력 시점 — 서버에서 실제 net 값 보정

어드민 매출 카드에서 매출액 + 사은품 비용 저장될 때, 서버 라우트가 GA4 Measurement Protocol 로 두 번째 이벤트를 전송:

POST https://www.google-analytics.com/mp/collect
  ?measurement_id=G-XXXXXXXXXX
  &api_secret={API_SECRET}

{
  "client_id": "{consultations.ga_client_id}",    // 폼 제출 시 저장해둔 값
  "events": [{
    "name": "purchase",                // 또는 'lead_confirmed'
    "params": {
      "lead_id": "consult_abc123",    // 1번 이벤트와 동일
      "value": 47500,                 // 실제 net = 50000 - 2500
      "currency": "KRW",
      "transaction_id": "rev_xyz789"
    }
  }]
}
💡 client_id는 폼 제출 시 GA4 쿠키 _ga에서 떼서 consultations 에 같이 저장. 이래야 매출 보정이 폼 제출과 동일 사용자로 묶임. 안 묶이면 GA4 가 두 이벤트를 다른 사용자로 봄 → 어트리뷰션 깨짐.

10-B-4. Google Ads 측 보정 (Conversion Adjustments)

광고 ROAS 정확도를 GA4 너머 Google Ads에서도 보려면, Ads 의 "전환 조정" API로 net 값을 한 번 더 보정.

  1. 폼 제출 시 gclid 캐치 (이미 어트리뷰션 시스템에 있음) → consultations 저장
  2. 매출 입력 시 서버에서 Google Ads API conversionAdjustments.upload 호출 → 해당 gclid의 전환값을 net 마진으로 보정
  3. Ads 화면에서 "전환 추적상태"가 "조정됨"으로 바뀌면 OK
⚠️ Google Ads Conversion Adjustments API 는 OAuth 2.0 + 개발자 토큰 신청 필요. 1차 런칭 때는 GA4 Measurement Protocol 만 우선 구현 → Ads-GA4 import 로 ROAS 받고, 트래픽 안정화 후 Ads API 직접 보정 도입 권장.

10-B-5. 메타·네이버 광고는?

매체실 매출 보정 방법
Meta (페이스북) Conversions API (CAPI) — 서버에서 직접 Purchase 이벤트 + value 보냄. 픽셀의 Lead 이벤트와 함께 dedup.
네이버 검색광고 전환값 직접 보정 API 미지원. 전환별 평균값을 운영하면서 매월 보정 (네이버 광고 시스템 → 전환 → 가치 갱신)
네이버 GFA 동일. 평균값 수동 보정 운영.
카카오 모먼트 동일. 평균값 수동 보정.

10-B-6. 구현 체크리스트 (코드 작업 단계)

  1. consultations 테이블에 ga_client_id, ga_session_id, gclid, fbclid 컬럼 (어트리뷰션 hook 일부 이미 있음 — 점검)
  2. ApplyForm 제출 시 GA4 쿠키에서 _ga 파싱 → 서버에 같이 보냄
  3. 폼 제출 성공 응답 후 dataLayer generate_lead push (lead_id + 추정 value)
  4. /api/admin/revenue POST 시 서버에서 net_value = amount − COALESCE(gift_cost, 0) 계산 → GA4 Measurement Protocol purchase 이벤트 전송
  5. Vercel env 추가: GA4_MEASUREMENT_ID, GA4_API_SECRET
  6. (선택) Google Ads Conversion Adjustments API 연결 — 트래픽 안정화 후
✅ 이 매핑이 끝나야 진짜 매체별 ROAS가 보입니다. "naver-cpc 평균 LTV 47,500원 / 메타 32,800원" 식으로. 매체 예산 분배 의사결정의 기반.

11. 검증 체크리스트

11-1. Chrome 확장 3종 설치

  • Tag Assistant Legacy (by Google) — GTM·GA4·Ads 태그 발사 확인
  • Meta Pixel Helper — 메타 픽셀 발사 + payload 확인
  • GA Debugger — GA4 hit 콘솔 로그 출력

11-2. GTM Preview 모드

GTM 우상단 "미리보기" → 사이트 새 탭 → 좌측 Tag Assistant 패널에서:

  • Tags Fired: 발사된 태그 목록
  • Tags Not Fired: 트리거 조건 미충족 (의도된 거면 OK, 아니면 트리거 점검)
  • Variables: 각 이벤트 시점의 dataLayer · GTM 변수 값
  • Data Layer: 시간순 push 기록 (이벤트 이름·payload 그대로 보임)

11-3. GA4 DebugView

GA4 → 관리 → DebugView. 실시간 이벤트 흐름 + 매개변수 확인. generate_lead가 보여야 광고 ROI 측정 가능.

11-4. Ads 전환 진단

Google Ads → 도구 → 전환 → 상태 컬럼. 폼 1건 제출 후 ~24시간 내 "기록 중"으로 바뀌면 OK.

12. 광고 ON 직전 점검 리스트

실제 광고비 태우기 직전, 아래 12개 모두 ✅ 인지 점검:

  1. GTM GTM-N3HSNZPJ 컨테이너 사이트 박힘 확인 (Tag Assistant)
  2. GA4 측정 ID 연결 + DebugView에서 page_view 흐름 확인
  3. GA4 "이벤트" 메뉴에서 generate_lead 를 전환으로 표시 ON
  4. Google Ads 계정 ↔ GA4 링크 (전환 import용)
  5. Google Ads 전환 액션 1건 이상 활성 + GTM 전환 태그 트리거 generate_lead
  6. Meta 픽셀 PageView + Lead 둘 다 GTM Preview에서 발사 확인
  7. 네이버 광고 픽셀 (검색 + GFA) 발사 확인
  8. GSC + 네이버 SC 인증 완료 + sitemap 제출
  9. UTM 표준 가이드 광고대행사 핸드오프 완료 (/admin/help/utm)
  10. dataLayer generate_lead push 실데이터 1건 시뮬레이션 (테스트 신청 폼 1회 제출)
  11. 전환가치 동적 매핑: 매출 입력 시 서버에서 net 마진(매출−사은품)을 GA4 Measurement Protocol로 보정 (10-B 섹션)
  12. 슬랙 알림 시스템 ON — 신규 디비 알림 + 이상 시그널(CPA 2배수 등)
  13. ad_metrics 시트 sync 1회 실제 데이터로 검증 + 어드민 대시보드 매체별 ROAS 표시 확인
✅ 12개 모두 OK = 광고 ON 가능. ROI 측정·이상 감지·매체별 분석 인프라 완비.
⚠️ 1번부터 6번까지(측정·전환)는 광고 시작 전 무조건 끝나야 합니다. 7~9번(GSC/SC/UTM)은 광고와 무관하게 늘 OK여야 합니다. 10~12번은 광고 시작 첫 주 안에 안정화하세요.
