## 1. 광고 시스템 의사결정

### 1.1 [2026-05-27] 매체값 정규화 — sheet_channel_alias 별도 테이블

**상황**: 시트 매체값('네이버 검색광고', '메타 비즈니스' 등)이 한글이라 `channel_mapping` (utm_source/medium 기반) 과 직접 매핑 안 됨.

**옵션**:
- A. channel_mapping 의 utm_source 에 한글값 추가
- B. **별도 테이블 sheet_channel_alias 신설** ← 선택
- C. 코드에 하드코딩

**결정**: B (별도 테이블)

**이유**:
- channel_mapping 은 utm 매핑용 (consultations 측)
- 시트 매체값은 사이트 utm 과 별개 (광고비 측)
- 어드민에서 편집 가능하게 (향후 매핑 페이지 추가 시)
- 새 매체 추가 시 SQL 없이 어드민에서

**구현**: `20260527040000_sheet_channel_alias.sql`

---

### 1.2 [2026-05-27] 우리편 vs 오즈랩 분리 — site 컬럼

**상황**: 우리편 4-5월 데이터 import 필요. 오즈랩 본격 운영과 분리해야.

**대웅 답변**: "4~5월 데이터는 오즈랩이 아니라 우리편 데이터네. 일단 데이터 넣어두고, 실제 오즈랩이 빌드되면 기존 데이터는 삭제해야해."

**옵션**:
- A. 별도 테이블 (ad_metrics_wooripen)
- B. **ad_metrics.site 컬럼 추가** ← 선택
- C. channel 에 prefix (wooripen-naver-search)

**결정**: B (site 컬럼)

**이유**:
- 한 테이블에 다 들어가서 쿼리 통일
- 멀티테넌트 (SaaS) 확장 시 그대로 활용
- `DELETE WHERE site='wooripen'` 한 줄로 정리

**구현**: `20260527050000_ad_metrics_site.sql` + unique 인덱스 (site, date, channel, service)

---

### 1.3 [2026-05-27] 광고측 vs CRM측 리드 분리

**상황**: 우리편 데이터에서 매체별 "리드" 컬럼이 모두 0 (consultations utm 매칭 안 됨).

**대웅 답변**: "컬럼 2개 분리 (투명)" + "전환도 동일 로직 적용" + "전체 site 동일 로직 (추천)"

**옵션**:
- A. MAX(utm 매칭, ad_metrics.conversions)
- B. **컬럼 2개 분리** (광고측 리드 + CRM측 리드) ← 선택
- C. ad_metrics.conversions 는 fallback 으로만

**결정**: B (분리 표시)

**이유**:
- 투명성 — 어트리뷰션 갭 한눈에 보임
- 광고 플랫폼 vs CRM 도착 정의가 다름 (광고 클릭 후 폼 미제출 → 갭 발생)
- 우리편 데이터 + 오즈랩 데이터 모두 의미 있는 표시

**구현**:
- `lib/admin/paid-media.ts` 의 `ChannelPerformanceRow` 에 `ad_leads`, `ad_cpl` 추가
- 매체별 표 14컬럼 (보라 = 광고측, 블루 = CRM측)
- KPI 카드 10개로 확장

---

### 1.4 [2026-05-27] (date, channel, service) 사전 집계

**상황**: 시트 _서비스분류 탭에 같은 (date, channel, service) 키에 여러 캠페인 raw 행 (평균 14개). upsert 시 마지막 행만 남는 문제.

**옵션**:
- A. ad_metrics 스키마 변경 (campaign 컬럼 추가 + unique 확장)
- B. **사전 집계 후 upsert** (SUM) ← 선택

**결정**: B (사전 집계)

**이유**:
- 캠페인별 raw 보존 안 되지만 paid-media 페이지의 캠페인 드릴다운은 consultations.utm_campaign 기반이므로 영향 없음
- 스키마 변경 최소화
- 향후 캠페인 raw 필요해지면 ad_metrics_raw 신규 테이블 검토

**구현**: `app/api/admin/ad-sync/route.ts` 의 `syncOneSheet` 에 `aggMap` 추가

## 2. 상품 시스템 의사결정

### 2.1 [2026-05-27] 통합 양식 1개 vs 양식 종류별 분리

**상황**: NIT 출고정책 + 네이버 렌탈표 + 해피 SKT정책서 등 공급사별 양식이 다 다름.

**대웅 답변**: "헤더통합하고 하나의 양식으로 해야해. 없는 헤더는 공란으로 두면 되도록. 헤더 이름은 우리 규칙대로 한글로, 누가봐도 쉽게 인지하고 사용할 수 있도록"

**옵션**:
- A. 양식 2~3종 (정찰제 + 렌탈 + 인터넷)
- B. **통합 양식 1개** ← 선택
- C. 양식 자동 감지

**결정**: B (한 양식)

**이유**:
- 담당자 헷갈림 최소화
- 공급사별 양식 헤더는 자동 인식 (bulk API alias)
- 인증 컬럼 같은 무관 컬럼은 자동 무시

**구현**:
- 표준 헤더 11개 (`상품 이름 / 분류 / 공급사 / 원가 / 우리 수당 / 고객 가격 / 약정 기간 / 월 정기 결제 / 월 결제 금액 / 단말기 종류 / 메모`)
- bulk API alias 풍부 (`pick(r, 'label', '상품 이름', '품목명', '상품구성', ...)`)

- bulk API 의 검증/매핑 로직 그대로 재사용

- 코드 중복 없음

**주의**: 향후 lib 분리하면 더 효율적 (현재는 OK).

## 9. 이 시스템 만들 때 자주 발생한 패턴

### 9.1 사용자 정정 → 메모리 반영

대웅이 정정해주는 경우 즉시 메모리에 박음 (예: `ozlab_product_vision` 의 4종 통합 + 4차원 비전).

→ 신규 작업 시작 전 `MEMORY.md` 확인 필수.

### 9.2 PRD 먼저 → OK → 코드

큰 작업은 PRD .md 먼저 작성 → 사용자 검수 → 코드.

→ `docs/PHASE_E_PRD.md` 가 좋은 예시.

### 9.3 dry_run 우선

시트 sync / bulk 모든 작업은 dry_run 옵션 보유. 사용자가 검증 후 실행.

→ Sync API 의 표준 동작 패턴.

### 9.4 친근한 반말 + 직설

대웅 user_preferences. 어드민 UX 메시지도 반말 톤. 단 가이드 문서는 존댓말 (담당자용).
